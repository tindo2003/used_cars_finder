# Skills to Learn

Compiled by reviewing all 84 commits in this repo (`git log --reverse --stat`). This is not a code review ŌĆö it's a map of the concepts underneath the code, for going back and actually learning them instead of just shipping them. Each item names the commit(s) where it showed up so you can `git show <hash>` and read the real diff while you study.

Organized by domain, roughly in the order you'd want to learn them (each section builds a bit on the one before), with a "start here" shortlist at the top.

## 1. Python backend engineering

- **Dataclasses vs. plain dicts/kwargs** (`16b8b3e` "Replace per-provider positional args with a shared ScrapeOptions dataclass"). You had 4 scraper providers each taking a slightly different pile of positional args ŌĆö classic sign a dataclass (or at minimum keyword-only args) was overdue. Learn: `@dataclass`, why it beats a dict for "a bag of related config," default values, and `dataclass(frozen=True)` for immutability.
- **Pydantic models & runtime validation** (`c2b1afd`). Learn the difference between Pydantic (validates real data at runtime, raises/reports on bad shape) and a `TypedDict` (compile-time only, zero runtime cost, no protection). You explicitly chose Pydantic here for a real reason ŌĆö make sure you understand *why* that reason mattered (bad scraped data silently propagating as wrong-shaped `None`s).
- **Static typing with mypy** (`c2b1afd`, `.github/workflows/typecheck.yml`). `disallow_untyped_defs` is a specific, learnable mypy setting ŌĆö look up what it does and doesn't catch. Try intentionally introducing a type error locally and reading mypy's error output; it's a skill in itself to parse.
- **argparse** (`scraper/main.py`, `scraper/notify.py` ŌĆö `--dry-run`, `--max-pages`, `--notify-top-n`, `--stale-threshold-days`). Small but everywhere. Learn `add_argument`, types, defaults, and why CLI flags beat hardcoded constants for a script you re-run with different parameters.
- **Class extraction / "separate orchestration from I/O"** (`8563651` ListingsDB, `2953e36` renamed to generic DbClient, `1d73b66` ScrapeRunner extracted from `main.py`). This is a specific refactoring skill: pull the part that talks to a real system (DB, network) behind a class with a narrow interface, so the *logic* around it becomes testable without that real system. Look at the diff in `1d73b66` specifically ŌĆö it's a clean, small example of this pattern.
- **Pure functions for testability** (`6d26b4a` "Extract pagination-stall detection into a tested pure function"). `page_did_not_advance()` takes plain data in, returns a plain bool ŌĆö no page object, no browser. This is *why* it got 52 lines of tests trivially. Compare to how much harder `providers/dealeron.py`'s `scrape()` itself is to test (it isn't, directly ŌĆö it needs Playwright). Learn to spot which parts of a function are "pure logic" vs. "talks to the world," and extract the former.

## 2. Testing

- **pytest basics**: fixtures, parametrization you haven't used yet, `assert` style vs. `unittest`'s `self.assertEqual`. You have ~140 tests and (from what's in this repo) don't seem to use fixtures or parametrize ŌĆö worth learning both, they'd shrink a lot of the repeated `make_listing(...)`/`make_search(...)` helper patterns in `tests/test_*.py`.
- **Test doubles: fakes vs. mocks** (`scraper/tests/fakes.py`, introduced `191e813`, grown steadily since). `FakeSupabase` is a *fake* ŌĆö a real, working, simplified implementation (in-memory tables with actual filter semantics) ŌĆö not a `Mock()` that just records calls. Learn the distinction and when each is appropriate; a fake is more work up front but catches real logic bugs a mock can't (a mock will happily "confirm" you called `.eq("status", "active")" even if your fake data doesn't actually respect that filter).
- **Frontend testing with Vitest + Testing Library** (`63c6368` onward, `app/page.test.tsx`). Specifically: `getByRole`/`getByLabelText` over `getByTestId` or CSS selectors ŌĆö this is "test like a user, not like the DOM," and it's why your multi-select dropdown needed a real `<label htmlFor>` association to be testable at all (you hit this directly this session). Learn `userEvent` vs. `fireEvent` (the former simulates real browser event sequences; the latter fires one raw event).
- **Testing a Playwright-driven function without a real browser** (`scraper/tests/test_dealeron.py`, this session). The `FakePage` stub only implements the 2 methods `_scroll_until_stable()` actually calls. This is the same "extract + fake" skill as above, applied to browser automation specifically.

## 3. Databases (Postgres / Supabase) ŌĆö the section you keep getting bitten by

- **Migrations as append-only history, run manually** (all `migrations/*.sql`, 14 of them). You've been doing this right (numbered, never edited after the fact) ŌĆö make sure you understand *why*: a migration is a record of what changed and when, and editing an old one after it's been run breaks that history for anyone re-applying from scratch.
- **Upsert conflict targets (`ON CONFLICT`)**. You've had this bite you three separate times:
  - `9703bff` ŌĆö dedupe key was originally `original_url`, switched to `vin`.
  - `24b3dd6`/`ecc1096`/`f5b280a`/`a179812` ŌĆö a multi-commit saga where `original_url`'s own leftover unique constraint kept crashing upserts keyed on `(vin, dealer_name)`, because two *different* unique constraints existed on the same table and only one was accounted for.
  - Learn: what `ON CONFLICT (columns)` actually requires (a matching unique constraint/index, not just "these columns look unique to me"), how to list a table's real constraints (`pg_constraint`, or `\d tablename` in `psql`), and why guessing a constraint's name (`24b3dd6`'s lesson) is fragile ŌĆö look it up by definition instead.
- **Column type migrations with `USING`** (migration 014, this session ŌĆö `text` ŌåÆ `text[]`). The `USING case when make is null then null else array[make] end` clause is doing real work: without it, Postgres won't know how to convert existing string data into the new array type. Learn `ALTER COLUMN ... TYPE ... USING`.
- **Row Level Security (RLS)** (`23afb44`, `ae7c6e9`, `4x_relax_rls...sql`). This is worth real study, not just "run the SQL Claude gave you": `auth.uid() = user_id` policies mean **the database itself** refuses cross-user reads, independent of any bug in your frontend code. Learn: policies are per-operation (select/insert/update/delete), `USING` vs `WITH CHECK`, and what happens when RLS is *on* but no policy matches (default-deny). Also worth understanding: migration 004's "no-auth stopgap" (anon key + `true` policies) ŌĆö that's RLS effectively turned off, and knowing when that's happening vs. not is a real security-relevant skill.
- **PostgREST query semantics via supabase-js** (`.eq`, `.in`, `.or`, `.ilike`, `.is`) ŌĆö these aren't arbitrary method names, they map to real Postgres operators, and picking the wrong one has real behavior differences (`.in()` is case-sensitive equality; `.ilike` without wildcards is case-insensitive equality ŌĆö you hit this exact gap this session with make/model casing). Worth reading PostgREST's actual filter operator docs once, rather than pattern-matching method names from examples.
- **Foreign keys + embedded resource selects** (`select("*, listings(*)")` in the Favorites feature, `97a35a4`). This only works because `favorites.listing_id` has a real FK to `listings.id` ŌĆö PostgREST uses the FK to know how to join. Worth understanding *why* this one query replaces what would otherwise be two round trips.

## 4. Auth & sessions

- **Supabase Auth: email/password vs. magic link vs. OAuth** (`2131aa6`). You picked email/password specifically to avoid a callback/redirect route. Understand the actual tradeoffs of each (not just "which was less code this time") ŌĆö magic link and OAuth both need a redirect URL registered and handled, which is real infra, not just a UI choice.
- **Next.js middleware + SSR cookie handling** (`bd37969`). This one's worth sitting with: the Supabase SSR helpers existed in the repo *before* auth was wired up, but nothing ever called `getUser()` from an actual `middleware.ts` ŌĆö so sessions would never have persisted across requests despite all the scaffolding looking correct. Learn how Next.js middleware runs on every request, and why refreshing an auth session needs to happen there (not just in client components).

## 5. Frontend (React / Next.js / TypeScript)

- **`useState` + `useEffect` + `useCallback` + `useMemo`, together** ŌĆö this is the core skill underneath basically every frontend commit, and where this session's multi-make/model work lived most heavily (`cb3594a`). Specifically worth understanding, since you'll hit it again:
  - Why `fetchListings` is wrapped in `useCallback` with a dependency array, and why a `useEffect` that just calls it fires again whenever those dependencies change (this is *why* your app live-searches as you type/click, with no explicit "submit" needed).
  - Why `availableModels` (this session) is `useMemo`'d instead of plain state ŌĆö it's *derived* from `makeModelPairs` + `make`, so keeping it as separately-set state would risk it going stale/out of sync. "Can I compute this from other state instead of storing it separately?" is a genuinely important React question.
  - The reconciliation effect that drops a stale model selection when make changes ŌĆö this is the "derived state needs to stay in sync" problem in its most common real form.
- **Controlled inputs & accessible custom components.** Your `MultiSelectDropdown` needed a real `<label htmlFor>`/`id` pair to be reachable via `getByLabelText` in tests (and, not incidentally, by screen readers) ŌĆö a `<div>` styled to look like a label doesn't count. Worth learning the HTML label-association rules generally (`for`/`id`, or wrapping), since it affects both testability and real accessibility.
- **Next.js image config** (`next.config.ts`, `da3e3d0`/`7e2db82`). `next/image` requires allow-listing external image domains ŌĆö this isn't Next.js being difficult, it's a real security boundary (arbitrary external image URLs are a vector for SSRF-adjacent abuse if unrestricted).
- **Supabase JS client basics**: `createClient`, `.from(table).select()/.insert()/.update()/.delete()`, and the fact that these return `{ data, error }` rather than throwing ŌĆö meaning every call site needs to actually check `error`, which is a different error-handling discipline than try/catch.

## 6. Web scraping & browser automation (Playwright)

- **Headless browser automation fundamentals**: `page.goto`, `wait_until="networkidle"`, `query_selector`/`query_selector_all`, `.click(force=True)`, `wait_for_timeout` vs. `wait_for_selector`/`wait_for_load_state`. This whole domain (`scraper/providers/dealeron.py`, `dealerinspire.py`) is worth a dedicated pass through Playwright's own docs ŌĆö you're using maybe 15% of its API and re-deriving patterns (like the scroll-until-stable fix this session) that Playwright often has a built-in primitive for.
- **Detecting "did the page actually change" vs. "did the click not throw"** (`ade30a6`, `6d26b4a`). A `.click()` succeeding is not proof anything happened ŌĆö you found this the hard way (12 real vehicles scraped 20x over). This generalizes: in any automation, "the action didn't error" and "the action worked" are different claims, and you need a way to verify the second one independently (here: comparing VIN sets before/after).
- **Diagnosing client-side rendering behavior via DevTools JS, not guesswork** (this session's Lexus Stevens Creek investigation). The actual technique ŌĆö `performance.getEntriesByType('resource')` to prove *no new network request* happened while the DOM changed ŌĆö is a generally useful debugging skill for distinguishing "lazy-fetched from server" vs. "virtualized/progressively-rendered from already-loaded data." Worth practicing on other sites; it's the difference between guessing at a fix and knowing which fix applies.
- **robots.txt / crawl-delay compliance** (`e92f4bf`, `74444ae`, `research/scraping-etiquette.md`). Worth actually reading a `robots.txt` file by hand at least once (they're short) rather than trusting a library's interpretation ŌĆö you made a real, deliberate call to drop eBay entirely based on reading its policy directly.
- **Base64-decoded data attributes** (`dealeron.py`'s `extract_price`, decoding `data-pricelib`). A small, specific technique: sites sometimes embed structured data in an attribute as base64 specifically to make naive scraping harder (or just as an internal encoding choice) ŌĆö decoding it directly with a real key:value parser was clearly the right call, but recognize this pattern for next time (check `data-*` attributes for base64 before assuming a scraper needs OCR/rendering tricks).

## 7. Data modeling & product-thinking skills (the non-code half)

- **Designing a scoring heuristic and knowing its blast radius** (`c87ad25`, `research/deal-scoring-heuristic.md`). The "good deal" signal is intentionally simple (median of same make/model/year-band/mileage-band), but the real skill on display is *documenting what it doesn't account for* (trim, condition, private-vs-dealer) as explicitly-accepted limitations rather than silently shipping something that looks more authoritative than it is.
- **Why mixing heterogeneous populations breaks a median** (`210abc4`, this session). Private-party and dealer prices are structurally different distributions for "the same car" ŌĆö averaging/medianing across both produces a number that doesn't represent either group well. This is a real, generalizable statistics point (not scraper-specific): always ask "are these things I'm comparing actually drawn from one population?" before computing a summary stat across them.
- **Treating "can't judge" differently from "judged and it's fine"** (`ranking_key`'s fallback to price-only ranking when there aren't enough comparables). A small but real design principle: absence of a signal isn't the same as a negative signal, and conflating the two produces misleading confidence.
- **Staleness/recency as a proxy signal, with its assumptions stated explicitly** (`fb0201c`). "A listing the scraper keeps re-seeing is more likely still available" is a heuristic, not a fact ŌĆö and the known asymmetry (dealer full-rescan vs. Craigslist only-if-searched-for) was documented as an accepted gap rather than ignored. Worth internalizing this habit: state the assumption your heuristic rests on, in writing, next to the code.

## 9. Git / engineering hygiene

- **Atomic, revertable commits** ŌĆö you've mostly done this well throughout the history (small, focused commits with clear messages), which made today's "make CLs as small as possible" request straightforward. Worth noticing what made it easy: files naturally grouped by concern (a migration, a provider, a test file) rather than one commit touching unrelated areas.
- **Root-causing instead of catching-and-suppressing** (`f5b280a` ŌåÆ `ecc1096` ŌåÆ `24b3dd6` ŌåÆ `a179812`, the `original_url` saga). The *first* fix caught the crash and skipped the row ŌĆö which quietly broke a different feature (duplicate detection) in a way that took another investigation to find. The eventual real fix (drop the redundant constraint, dedupe explicitly) is the better lesson than the quick patch. When a `try/except` makes a symptom go away, ask what else was depending on the thing you just suppressed.
- **A living decisions doc that survives across sessions** (`9a22f87` introducing `memory.md`, then continuously updated). This is a project-management skill as much as an engineering one: writing down *why* a decision was made (not just what changed) so a future session ŌĆö or future you ŌĆö doesn't re-litigate it from scratch or, worse, "fix" something that was intentional.
