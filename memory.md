## What this is

**Used Car Finder** — a personal tool (single account, email/password auth as of 2026-07-20) that scrapes used-car listings from Bay Area dealer sites and Craigslist, lets you define saved searches (make/model/year/mileage/price), and emails you a daily digest of new matches, ranked by an actual "good deal" signal rather than just filter-matching. Audience/vision: `research/prd.md`. Full status snapshot with `[x]`/`[ ]` per feature and the suggested build order: `research/mvp-checklist.md` — **read that one first** when picking up work.

## Key decisions already made (don't re-litigate without a reason)

- **eBay is intentionally not scraped.** Its `robots.txt` explicitly disallows the search pattern needed and states automated access is prohibited without permission. `providers/ebay.py` still exists for a possible future pass against their official Browse API, but is not in `main.py`'s `ACTIVE_MARKETPLACES`. Detail: `research/scraping-etiquette.md`.
- **Notifications run once daily, not every 15 min.** Explicit user choice to avoid frequent emails, even though the PRD says "prioritize speed over batching." This supersedes that PRD line for the current single-user deployment. Notification-checking is a separate GitHub Actions workflow from scraping (lighter, no Playwright, not delayed by slow scrapes).
- **The daily digest always sends the current top N best matches, resends included (changed 2026-07-20).** Previously `notify_matches()` permanently excluded a listing from future digests once notified once (`notification_history`'s unique `(saved_search_id, listing_id)`), so an already-notified listing never re-surfaced even after a real price drop -- a gap external review flagged 2026-07-19. Explicit user call when presented with the fix options: rather than build price-drop-detection logic to selectively re-trigger, just always send today's top N ranked by `deals.ranking_key`, whether or not they were sent before ("i want a list of top n good deals, no matter if notified or not"). `notify_matches()` no longer reads `notification_history` to filter candidates; `notification_history` is kept **write-only as a log** (a real design fork resolved by explicit user choice, since dedup was its only prior purpose) for a possible future click-through-rate metric, so migration 013 dropped its uniqueness constraint to allow one row per digest send rather than one ever. Ranking still falls back to lowest-price for listings without enough comparables to score (another explicit choice: top-N-best-matches, not strictly-only-`is_good_deal`-listings) so a sparse search still gets a full digest rather than an empty or short one.
- **Multi-make/multi-model search + grouped notification digests, built 2026-07-20.** `saved_searches.make`/`model` (and the frontend filter state) moved from scalar to `text[]` (migration 014) so a search/saved search can watch several makes and models at once -- semantics are AND-of-two-OR-sets (`make` matches ANY selected make AND `model` matches ANY selected model), not literal separate searches per combination. Frontend input is a checkbox multi-select (`MultiSelectDropdown` in `app/page.tsx`), not free text -- options are the distinct make/model values currently in `listings`, deduped case-insensitively (dealer feeds aren't consistent about casing, e.g. "Toyota" vs "TOYOTA" both appear in production) while preserving one display casing per value. Because of that same casing inconsistency, the frontend's listings query matches with `ilike` (case-insensitive exact match, no wildcards) via `.or()` rather than `.in()`, which is case-sensitive and would have silently missed casing variants -- found and fixed during live verification against production data. `scraper/notifications.matches()` mirrors this: any-of-list, case-insensitive exact match for both make and model (model dropped its old substring/`ilike` matching, since the multi-select now offers exact distinct values, not free-typed fragments). Saved searches also gained `notification_grouping` (`combined` default / `make` / `model`): grouped digests split matches by the listing's actual make/model value and send one email per group, each independently capped to `top_n` (not a shared top_n split across groups) -- confirmed live: a real digest run against production produced 2 separate emails for a Toyota+Lexus search grouped by make. **Digest email subject/intro now identifies the search (fixed 2026-07-20, same day, after user feedback from real production emails).** `notify_matches()` passes a `search_label` through to `send_digest_email()` (`_search_label()` in `scraper/notifications.py`): prefers the saved search's `name` (quoted), falls back to describing its make/model filters, else `None` (generic "your saved search", only for a fully unfiltered search). Combines with `group_label` when both apply (e.g. "Top 10 Toyota matches for 'SUV hunt'"). `send_email_fn` callers only get `group_label`/`search_label` kwargs when non-`None`, preserving backward compatibility with simpler callers/tests.
- **Frontend UX pass, built 2026-07-20 (same day as multi-make/model): searchable dropdowns, dependent model options, "Load More" pagination, collapsible sections.** `MultiSelectDropdown` gained a small text filter (client-side substring match on the already-fetched options, not a live query -- still no autocomplete) so a long option list (every make in production) is searchable instead of requiring scrolling. Model options now narrow to whatever models actually belong to the selected make(s) (`availableModels` derived via `useMemo` from raw `{make, model}` pairs, not a flat deduped list) -- a `useEffect` prunes a previously-selected model out of `model` state if it falls outside the newly-narrowed list, so a stale filter never stays silently applied. Separately, the listings query was hardcoded to `.limit(50)` with no way to see anything past that (never previously flagged as a gap) -- replaced with `.range()`-based "Load More" pagination via a shared `buildListingsQuery()` (both the initial fetch and Load More apply a different `.range()` to the same filtered/sorted query); added `id` as a final sort tiebreaker so pagination stays deterministic even when price/`last_seen_at` both tie across a page boundary. "My Saved Searches" and "My Favorites" section headers are now toggle buttons (count + Hide/Show), both defaulting expanded. Verified live against production: in-dropdown search filters correctly, picking "Toyota" narrows Model to only Toyota's models, and Load More doubled the rendered card count (50→100) with no new page-boundary duplicates.
- **`scraper/runner.py`'s dynamic Craigslist search broke in production from migration 014, then the underlying gap was closed for good (both 2026-07-20, same day).** `run_marketplace_searches()` (drives one Craigslist search per active saved search, so newly-searched-for make/models get picked up even between full dealer re-scrapes) still read `search.get("make")`/`.get("model")` as scalar strings and called `.capitalize()` directly on them -- crashed with `AttributeError: 'list' object has no attribute 'capitalize'` the first time a real GitHub Actions run hit a saved search with the now-array `make`/`model` columns. Missed originally because the multi-make/model migration's review only checked `scraper/notifications.py`'s usage, not `runner.py`'s -- and `runner.py` took `saved_searches` as untyped `Dict[str, Any]`, so mypy had nothing real to check `.capitalize()` against. First fixed the immediate crash (iterate every make x model combination per search, same AND-of-two-OR-sets semantics as everywhere else -- `[None]` stands in for an unset dimension so a single-sided filter still runs once, not zero times; a fully unfiltered search is still skipped entirely, unchanged). **Then closed the systemic gap the user asked for ("make this not happen again"):** `main.py` now validates `saved_searches` rows through `SavedSearch.model_validate()` before they ever reach `runner.py`, and `ScrapeRunner.run_marketplace_searches`/`.run()` are typed to take `List[SavedSearch]`, not raw dicts -- reversing part of the original Pydantic scoping decision below, which explicitly excluded `runner.py`. This means the *next* `SavedSearch` schema change gets caught by mypy at every call site automatically, not just the ones a human remembers to update by hand. Also fixed `main.py`'s `--dry-run` dummy fixture, which had independently drifted to the old scalar shape (`{"make": "Toyota", ...}`) -- a second, separate instance of the same "stale fixture masks a real schema mismatch" failure mode, meaning `--dry-run` testing locally wouldn't have caught this bug either. 2 new tests in `tests/test_runner.py` for the multi-value cross-product; all existing dict fixtures converted to real `SavedSearch` instances via a local `make_search()` helper (mirrors the one already in `tests/test_notifications.py`).
- **Transmission + seller-type filters, built 2026-07-20.** PRD items 4.2 that were previously deferred as lower-priority. Turned out to be a frontend-only addition -- `SavedSearch`/`notifications.matches()` already had scalar `transmission`/`seller_type` fields and matching logic, and a live read confirmed those columns already existed on `saved_searches` in production (predating the migrations/ convention, like `user_id` did) -- no migration needed. Options are built from real distinct values in `listings` (same `dedupeCaseInsensitive` pattern as make/model), not a hardcoded guess: Seller Type only ever offers "dealer" in practice (Craigslist/eBay don't expose an owner/dealer distinction at list level), and Transmission surfaced a real granularity problem -- dealer feeds store ~37 distinct manufacturer-specific strings ("6-Speed Automatic ECT-i", "9-Speed 948TE Automatic"), not a clean Automatic/Manual split. Transmission options narrow to the selected make(s)/model(s) (extending the same `filterOptionRows` -> `useMemo` narrowing pattern Model-under-Make already used), with the same "clear the filter if it falls out of the now-relevant list" pruning. Added an "Any Automatic" quick-pick (`ANY_AUTOMATIC_VALUE` sentinel, matched via `ilike("transmission", "%automatic%")`) so a customer doesn't need their car's exact manufacturer spelling -- deliberately literal, not semantic, by explicit user choice: CVT ("CVT Lineartronic", "eCVT") and dual-clutch ("PDK", "DCT") transmissions are functionally automatic (no clutch pedal) but stay outside the bucket since substring-matching "automatic" doesn't catch them and broadening the definition was explicitly declined. Also added a close ("x") button inside each multi-select dropdown panel so Make/Model don't require re-clicking the toggle to dismiss.
- **"Currently monitoring" live stats widget, built 2026-07-20 (a "just for fun" ask, not a PRD item).** Shows distinct source count, active listing count, listings created today, and time since last crawl on the homepage. Deliberately polling (30s interval), not Supabase Realtime/websockets -- the underlying data only changes when the scraper cron runs (every 15 min), so a websocket wouldn't make this meaningfully more real-time, just more infrastructure (Postgres replication, subscription/reconnect lifecycle) for a personal single-user app where nobody else needs updates pushed to them. The "live" feel comes from a separate 1s UI tick recomputing the relative-time display between polls, not from more frequent data fetches. One lightweight query backs all four stats (id/created_at/last_seen_at/dealer_name/marketplace_source for active, non-duplicate listings), derived client-side -- same pattern as the filter-option dropdowns. `formatLastSeen`'s relative-time formatting was extended to second-level granularity (was minutes-only) so "last crawl" can read like a real clock. Widget hides itself entirely when there's nothing to report rather than showing zeros.
- **Saved-search edit + pause/resume, built 2026-07-20.** Edit reuses the existing main search form rather than a second UI surface: clicking Edit populates make/model/sliders/name/email/grouping from that saved search and flips `handleSaveSearch` from an `insert` to an `update` keyed by `editingSearchId` (cleared -- along with the rest of the form -- only after the success/error status window closes, so "Updated!" has time to actually display instead of being immediately overwritten by the create-mode "Saved!" text). Pause/Resume just flips the existing `is_active` column (already read by `notify_matches()`'s `is_active=True` filter -- no migration needed) and shows a "(Paused)" label.
- **Next.js image optimization disabled for hotlinked photos (fixed 2026-07-20).** Images stopped rendering; `next/image`'s optimizer proxy was returning 400s for every dealer/marketplace photo host. Root cause (confirmed via server logs): its SSRF guard refuses to fetch an upstream image if the hostname resolves to what it flags as a "private" IP, and every photo host was resolving through a NAT64 gateway (`64:ff9b::...` synthetic addresses) -- a false positive, and a real-world failure mode for anyone on an IPv6-only network/mobile carrier, not unique to this dev sandbox. Since `next.config.ts`'s `remotePatterns` already allows every hostname (`"**"`), the optimizer wasn't narrowing any real security surface here anyway, so `images.unoptimized: true` sidesteps the whole class of failure by serving original URLs directly. Confirmed live: images render again, zero `/_next/image` requests fired.
- **Auth: email + password, built 2026-07-20.** Chosen over magic-link specifically because it needs no callback/redirect route -- `signUp`/`signInWithPassword` return a session directly. Requires "Confirm email" disabled in the Supabase dashboard (done) so there's no confirmation-link redirect to handle either. Saved Searches is now scoped to real `user_id` + RLS (migration 010, `auth.uid() = user_id`), replacing the old no-auth localStorage stopgap (migration 004) entirely -- that mechanism is fully removed from the frontend, not just deprecated. Along the way, found and fixed a real gap in the pre-existing Supabase SSR scaffolding: `utils/supabase/middleware.ts` built cookie handlers but never called anything to trigger them, and nothing invoked it from an actual Next.js `middleware.ts` (which didn't exist) -- so sessions would never have persisted across requests. Verified live end-to-end against production: sign-up gives an immediate session, it survives a reload, a saved search is correctly scoped to the account, log-out hides it, log-in restores it; test account and data cleaned up after.
- **Gap in that same migration, found and cleaned up 2026-07-19: RLS-scoping migrations must explicitly handle pre-existing rows, not just new ones.** Migration 010 added `auth.uid() = user_id` RLS to `saved_searches` but never backfilled the 2 rows that predated real auth (from the migration-004 no-auth stopgap) -- they kept `user_id = NULL`. RLS correctly hides those everywhere (`NULL` never equals a UID), making them permanently invisible/unmanageable from the UI (can't see, edit, pause, or delete). But `notify.py` reads via the service-role key, which bypasses RLS entirely, so it kept finding and emailing both orphaned rows regardless -- surfaced as the user getting an unexplained extra digest email from a saved search they couldn't see anywhere. Looked like a scraper bug; was actually stale data from an incomplete migration. A full-table check found no other rows in this state, and both were deleted after confirming with the user. Full writeup: `research/data-quality-findings.md`. **Rule for any future migration that adds RLS scoping to an existing table**: immediately query for `<new_scoping_column> IS NULL` and explicitly decide what happens to those rows (backfill to a real owner, or delete) as part of the migration, not as an afterthought.
- **Favorites, built 2026-07-20.** RLS scoped the same way (migration 011, `auth.uid() = user_id`, plus a unique index on `(user_id, listing_id)` so a listing can't be favorited twice -- `favorites` predates the migrations/ convention like `saved_searches` did, so its RLS state needed the same explicit treatment migration 010 gave that table). Heart toggle on every card, only shown when logged in; a "My Favorites" section stays a section on the same page rather than a new route, consistent with this app's single-page frontend. Favorited listings fetch in one query via PostgREST's embedded-resource select (`select("*, listings(*)")`) -- confirmed `favorites.listing_id` has a real FK to `listings.id` before relying on this. Verified live end-to-end the same way Auth was: favoriting/un-favoriting a real listing, test account and data cleaned up after.
- **The "good deal" signal is deliberately crude.** Compares a listing's price to the median of other active listings with the same make/model **and same `seller_type`** within ±2 model years and ±20,000 miles (needs ≥3 comparables to trust the median); ≥12% below median counts as a deal. No trim-level distinction, no condition/accident history. Full rationale and known limitations: `research/deal-scoring-heuristic.md`.
- **Comparable pool now separates dealer vs. private-party listings (fixed 2026-07-20, same day as multi-make/model).** `deals._is_comparable()` requires matching `seller_type` (case-insensitive; Craigslist's `None` matches `None`, so it still compares against itself, just not against dealer rows). Root cause this fixes: real production evidence (a Lexus-grouped digest dominated by very old, $2,900–$6,800 Craigslist listings) showed mixing channels let a cheap-by-channel private-party listing look like an outsized "deal" against a dealer-inflated median, or vice versa. Chose strict same-type-only exclusion over a mixed-pool fallback (both previously left open in `research/deal-scoring-heuristic.md`) so a thin per-channel pool falls back to price-only ranking rather than reintroducing the same skew through a fallback path -- consistent with the heuristic's existing "can't judge ≠ not a deal" philosophy. 2 new tests in `tests/test_deals.py`.
- **Lexus Stevens Creek's DealerOn listing page virtualizes rendering (diagnosed and fixed 2026-07-20).** GitHub Actions logs showed the scraper stopping after "Page 1" though the site reports 56 vehicles. Live inspection (browser JS, not the scraper) ruled out a broken selector -- both the "Show 96" dropdown and `.pagination__item--next` matched the real DOM fine. Real cause: cards for the CURRENT page mount progressively as you scroll, from data the site already fetched in one API call (confirmed: card count climbed 4→6→20→24 across repeated scrolls with *zero* new network requests, via `performance.getEntriesByType('resource')`) -- a single Playwright snapshot right after `networkidle` only ever captured whatever rendered first. Fixed with `providers/dealeron.py`'s new `_scroll_until_stable()`: scrolls to the bottom repeatedly (bounded, default 20 attempts) until the rendered `.vehicle-card` count stops growing, called once per page right before parsing. Costs ~0 extra overhead on non-virtualized dealers (stabilizes after 1 scroll). 3 new tests in `tests/test_dealeron.py` using a fake Playwright `page` stub (no real browser needed).
- **Dedup key is `(vin, dealer_name)` when vin present, manual lookup by `original_url` as fallback** (changed 2026-07-20 from bare `vin` -- see migration 007 and below). This only catches the same ad seen twice from the *same* dealer/source -- **cross-marketplace duplicate detection** (`scraper/duplicates.py`, built 2026-07-20) is the separate pass that catches the same vehicle posted on two different sources with no shared identifier: (a) a dealer's own site plus its own Craigslist repost -- fuzzy match on make/model/model_year + mileage within 500mi + price within $250 across differing `marketplace_source`; (b) an **exact VIN match**, unconditional and bypassing the fuzzy/marketplace_source checks entirely, added after a user-flagged gap -- dealer groups can syndicate the same VIN across sister storefronts (e.g. a trade-in shows up on both "Capitol Honda" and "Capitol Ford"), which used to be invisible: the old bare-`vin` upsert conflict key made the second store's scrape silently overwrite the first store's dealer_name/city/original_url in place, no duplicate row, no history, just a listing that flip-floped location depending on scrape order. Both paths write `duplicate_of` (migration 006) onto the non-canonical row daily via `notify.py`; frontend and `notify_matches` both exclude flagged duplicates.
- **`original_url` is NOT globally unique, and is no longer enforced as such (migration 012, 2026-07-20).** It has its own table-wide unique constraint since migration 001, back when it was the only conflict key -- that stuck around after migration 007 moved dealer rows to `(vin, dealer_name)`, and it doesn't actually hold: a dealer's own inventory card can legitimately link to a *different* dealer's page for the same syndicated vehicle (confirmed live: a Capitol Honda card linking to chevroletoffremont.com). The original fix (found 2026-07-20 from a production crash) caught the resulting `23505` violation in `db.upsert()` and skipped the row (`return None`) rather than crashing -- but that silently defeated `duplicates.py`'s exact-VIN-match pass, which exists specifically to flag two dealers sharing a VIN: the skip happened *before* the second dealer's row was ever inserted, so there was nothing left for that pass to find. (The earlier "0 of 740 listings shared a VIN" live check was measuring the wrong thing as a result.) External review caught this. Fixed properly (2026-07-20): migration 012 drops the constraint entirely (replaced with a plain non-unique index); `db.upsert()` no longer has a skip case at all -- dealer rows (have a `vin`) keep native `(vin, dealer_name)` upserts unchanged, and VIN-less rows (Craigslist, the only case where `original_url` was ever standing in for real identity) now dedupe via a manual lookup-then-write (`self.read(original_url=...)`, update if found, insert if not) instead of relying on Postgres's `ON CONFLICT` inference. Detail: `research/data-quality-findings.md`.
- **Pagination has a stall guard** (`pagination.page_did_not_advance`) after a real bug where "Next" clicks could succeed without the page actually changing, causing the same vehicles to be re-scraped up to hundreds of times in one run. Tested specifically to confirm genuine page-to-page advancing (including partial inventory overlap) is never mistaken for a stall.
- **Stale listings expire after a configurable window (default 90 days) of not being reconfirmed** (`scraper/staleness.py`, built 2026-07-20, user-directed design). `db.py`'s upsert stamps `last_seen_at` on every insert/update; `notify.py` runs `expire_stale_listings()` first (before dup/deal-score/notify steps) each day, marking `status="expired"` on anything past the threshold -- which then falls out of every other read for free (frontend, deals.py, duplicates.py, notifications.py all already filter on `status="active"`). Working assumption (user's, confirmed intentional): a listing the scraper keeps re-seeing is more likely still available than one it hasn't re-seen in months -- this is a proxy for "probably sold," not an actual delisting check (nothing verifies removal from the source). `last_seen_at` is also wired into `deals.ranking_key` and the frontend's listings query as a **tiebreaker only** (most-recently-seen wins ties), never the primary sort -- explicit user choice. **Known asymmetry:** far more reliable for dealer listings (full-inventory re-scrape every 15-min run) than Craigslist (`runner.py` only re-searches make/models with a currently *active* saved search, so a Craigslist listing can go stale just because no one's searching for it, not because it sold) -- user explicitly accepted this as an acceptable outlier rather than building Craigslist-specific compensation.
- **Dealer coverage**: 20 dealers currently in `main.py`'s `DEALERS` list (San Jose/Santa Clara/Sunnyvale/Fremont/Newark cluster). **`dealerdotcom` provider built 2026-07-22**, scoped to Mountain View/Sunnyvale/San Jose/Fremont per explicit user request -- every dealer in those 4 cities matching an already-built provider was already added in earlier passes, so this closed the remaining gap: Cox Automotive's Dealer.com platform, Akamai-fronted (confirmed *more* aggressive bot-wall than the Cloudflare gate `dealerinspire.py` handles -- a plain HTTP request, even to `/robots.txt`, gets `403` + `BOT-BROWSER-IMPERSONATOR`; a real Playwright browser session gets through fine using the same stealth launch args `dealerinspire.py` already established). Key discovery: these pages embed a full schema.org `CollectionPage` JSON-LD block per page with structured VIN/brand/model/year/price/transmission/fuel_type per vehicle -- far more reliable than the visible card (no VIN anywhere on-card except a flaky third-party badge, no clean make/model split). `dealerdotcom.py` joins each card to its JSON-LD entry by URL path. **One real trap found and worked around**: the JSON-LD's `mileageFromOdometer.value` is truncated to thousands (a real 32,373-mile car reports "32") -- mileage is deliberately still read from the card's own rendered badge, never JSON-LD, with a dedicated regression test pinned to this exact case. Unlocked 4 dealers at once, each verified live individually before adding (never bulk-added on a same-platform assumption): Premier Subaru of Fremont, Fremont Mazda, Honda of Stevens Creek, Stevens Creek BMW. Full writeup: `research/bay-area-dealer-candidates.md`. More candidates (including the still-unbuilt `dealereprocess` platform) with unverified platforms: same doc.
- **Pydantic + mypy, wired up 2026-07-20.** Prompted by a tangent about Python's lack of real enforcement behind `_helper` naming conventions (`scraper/` had zero type hints at the time). User explicitly chose Pydantic (real runtime validation) over a plain-`TypedDict` alternative (static-only, no runtime cost) specifically so bad scraped data gets caught the moment it enters the system rather than silently propagating as wrong-shaped `None`s into deal-scoring/notification logic. Scoped deliberately to contain the ~105-call-site blast radius: **validation happens only at the DB boundary** (`db.py`'s `bulk_save`/`upsert`/`read_listings`), not inside the scrapers -- `providers/*.py` still passes plain scraped-listing dicts around, converted to a validated `Listing` only right before/after touching Supabase. (`runner.py` was originally scoped out of this too, but that part of the decision was reversed 2026-07-20 after a real production break -- see the `runner.py`/`SavedSearch` entry above; `SavedSearch` is now validated before `runner.py` ever sees it.) `bulk_save` skips (doesn't crash on) any car that fails `Listing` validation, logging it and counting it in `progress["invalid"]`. `mypy` runs in CI (`typecheck.yml`, own workflow since it needs no Playwright) with `disallow_untyped_defs` on every source module; verified clean, plus the full 133-test suite passing and all 763 real production listings re-validating with 0 failures through `read_listings()` (read-only check, no writes).
- **Deployed to AWS, 2026-07-20 (ECS Fargate for scrape/notify, App Runner for the frontend) -- full detail in `terraform/README.md`, including a real first-deployment log with every CLI command.** Terraform-managed, GitHub Actions deploys via OIDC (no stored AWS keys). One real gap the deploy itself surfaced: `terraform apply` doesn't succeed unconditionally when the referenced `:latest` image doesn't exist yet -- ECS task definitions register fine regardless, but `aws_apprunner_service` validates the image synchronously at creation and hard-fails (`CREATE_FAILED`) if it's missing, corrected in the README's bootstrap order.
- **A real (non-dry-run) scrape/notify run never exits the process on its own -- found 2026-07-20 on the first real ECS Fargate run, full writeup in `research/data-quality-findings.md`.** The scheduled task logged its final "Done." line and then hung indefinitely (confirmed: a manual `aws ecs stop-task` succeeded with a clean `exitCode: 0`, meaning the work had genuinely finished) -- on a schedule this means indefinitely-billed Fargate time every 15 minutes, not just one slow run. Root cause is almost certainly a non-daemon thread the Supabase client leaves running (its `realtime`/`websockets` dependency is the prime suspect); CPython won't exit until every non-daemon thread finishes, and `sys.exit()` doesn't override that. **Never caught locally because `--dry-run` never calls `get_supabase()` at all** -- the one code path every real production run actually takes had never been exercised end-to-end before going live. Fixed with an explicit `sys.stdout.flush()` + `os._exit(0)` at the end of both `main.py` and `notify.py`'s `__main__` blocks. **Lesson: a `--dry-run`/mock code path that skips the real external client entirely can hide process-lifecycle bugs that only show up on a real run** -- worth remembering for any future addition of a new external SDK/client to this codebase, not just Supabase. **Re-verified on Fargate itself 2026-07-20 (a first attempt was inconclusive due to concurrent tasks from schedules firing mid-test; a clean single-task re-run afterward confirmed `STOPPED`/`exitCode: 0` with no manual kill needed)** -- both EventBridge schedules are back to `ENABLED` after being manually disabled during the investigation.

- **Listings geocoded into `listings.location`, built and backfilled live 2026-07-21 -- geocoding only, radius search filter itself deliberately deferred (explicit scoping decision).** `listings.location` and `saved_searches.target_location` turned out to already be live `geometry(Point,4326)` columns (confirmed via the Supabase REST OpenAPI schema, not previously captured in any tracked migration -- migration 015 now documents this ad-hoc-via-dashboard state, same treatment migration 010 gave `saved_searches.user_id`). New `scraper/geocoding.py` module: `geocode()` wraps a single Nominatim/OpenStreetMap `/search` call (free, no API key -- chosen given trivial call volume, only ~50 distinct city strings across all listings), `geocode_listings()` is a `staleness.py`-style enrichment pass (reads active listings via `read_listings()`, filters in Python to city-set-but-location-NULL since `DbClient` has no `IS NULL` filter, writes back via a plain `DbClient.update()` using EWKT text (`SRID=4326;POINT(lng lat)`) -- PostgREST accepts this directly on write, no RPC needed). Wired into `notify.py` right after `expire_stale_listings()`. Distinct city strings are cached per-run only (no persisted cache) since `notify.py` runs once daily, making even a full from-scratch re-geocode of all ~50 strings trivial Nominatim traffic. Verified live: backfilled 980 of 2052 active listings; the other ~1072 are Craigslist rows whose scraped "location" field is garbage (dealer phone numbers/CTAs like `"Call 408-539-3211"`, or Craigslist's own slash-separated multi-neighborhood text like `"dublin / pleasanton / livermore"`) that Nominatim can't resolve -- these correctly stay NULL rather than crashing the batch or writing nonsense. Sampled the 5 real dealer cities' resulting coordinates directly against Supabase to confirm correct placement and lat/lng ordering (the easiest bug to introduce here, since Nominatim returns lat/lng but PostGIS wants lng/lat). **Explicitly out of scope for this pass, follow-up work not an oversight**: the frontend "search near X, within Y miles" UI, a `ST_DWithin`-based RPC wired into `buildListingsQuery`, geocoding `saved_searches.target_location`, and wiring `search_radius_miles` into `notifications.matches()` -- `SavedSearch.target_location`/`search_radius_miles` remain unused stub fields. **Correction (2026-07-21, see the incident entry below): the `Optional[str]` typing claim in this entry was wrong.** This Supabase project actually serializes geometry columns as a GeoJSON dict, not EWKB hex text -- the assumption here was never verified against real data at the time, and the wrong type shipped anyway since nothing had yet tried to validate a populated (non-NULL) `location`/`target_location` value through Pydantic. Both fields are now `Optional[Dict[str, Any]]`.

- **Radius search filter itself, built 2026-07-21 (same PR/branch as the geocoding above -- `geocode-listings-radius-search-phase-1`).** Location input is a **zip code**, not a city picker or free-text address -- deliberately, to avoid needing frontend-side geocoding of arbitrary text. `migrations/016` adds `nearby_listings(center_lat, center_lng, radius_miles)`, a Postgres function using `ST_DWithin` (cast to `geography` for real-world meters, since `location` is `geometry`) that returns `setof listings` -- this matters because it lets `buildListingsQuery` chain its existing `.eq/.is/.order/.range` filters directly onto `.rpc("nearby_listings", ...)` exactly like it already does onto `.from("listings").select("*")`, so no filtering logic is duplicated between the RPC and the frontend. Also adds the GIST spatial index migration 015 deliberately deferred. Zip-to-coordinates resolution goes through **this app's first-ever server-side Next.js Route Handler**, `app/api/geocode-zip/route.ts` -- previously 100% of this frontend was client-side Supabase calls; a route was necessary here because Nominatim requires a real `User-Agent`, which isn't reliably settable from a browser `fetch`. `buildListingsQuery` itself stays synchronous -- the async zip lookup happens once, upstream, in `handleSearch`, producing `resolvedLocation` state that the query builder just reads like any other filter; deliberately does *not* call `fetchListings()` immediately after resolving (that would close over stale pre-update state) and instead relies on the existing `useEffect` that already re-fetches whenever `buildListingsQuery`'s dependencies change, same mechanism every other filter already uses. Verified live end-to-end: a real zip resolves to correct coordinates, the RPC call carries every chained filter correctly, results are all genuinely within the chosen radius, "Load More" reuses the resolved coordinates (confirmed via the browser's Performance API: exactly 1 `/api/geocode-zip` call across an initial search plus a Load More click), a malformed zip never touches the network, clearing the zip falls back to the plain query, and the anon/publishable key (what the frontend actually uses, not the service-role key) can call the new RPC without needing an explicit `grant`. **Explicitly still deferred**: saved-search radius and `notifications.py` matching -- the daily email digest does not respect radius, by explicit choice, same as the geocoding pass above.

- **Production incident: `notify.py` crashed on every run for ~24 hours (2026-07-21 15:00 UTC through the fix later that day) -- root cause was the wrong-type bug in the geocoding entry above.** `Listing.location`/`SavedSearch.target_location` were typed `Optional[str]` on an unverified assumption that PostgREST serializes `geometry` columns as EWKB hex text; the real value is a GeoJSON dict (`{"type": "Point", "coordinates": [lng, lat], ...}`). This was silent until the geocoding backfill actually populated real (non-NULL) `location` values -- the very next `notify.py` run (both the legacy GitHub Actions `notify.yml` cron and the EventBridge/Fargate `used-cars-finder-notify` task, which run the identical script) crashed immediately at `expire_stale_listings()` -- the first line of the pipeline -- with `pydantic_core.ValidationError: Input should be a valid string ... input_type=dict`, confirmed directly from the real CloudWatch log. Net effect: no digest email sent and no staleness/duplicate/deal-score updates applied for a full day, until caught (not by any alert -- found by inspecting the GitHub Actions run history while researching an unrelated feature request). **Fix**: both fields changed to `Optional[Dict[str, Any]]`; GeoJSON turned out easier to consume than EWKB hex would have been (`coordinates[0]`/`[1]`, no parsing library needed). **Lesson for next time a PostgREST/Supabase column's wire-format type gets assumed rather than verified**: validate the assumption against a real populated row before shipping, not just against a NULL one -- the phase-1 work above added the `location` field and ran `geocode_listings()` successfully, but that success only ever exercised the NULL case (checking `is None` before every write); nothing re-read a populated value back through `Listing.model_validate()` until a completely different, unrelated code path (`notify.py`'s other steps, which all call `read_listings()`) did so days later in production.

- **Saved-search radius + `notifications.py` matching, completed 2026-07-21 (same fix/PR as the incident above)** -- the piece deliberately deferred in the two entries above. Two new helpers in `scraper/geocoding.py`: `point_coordinates()` (GeoJSON dict -> `(lat, lng)`, flipping GeoJSON's `[lng, lat]` coordinate order) and `haversine_miles()` (great-circle distance, no new dependency). `notifications.matches()` gained one more filter block checking `search.target_location`/`search.search_radius_miles` against `listing.location`, following the file's existing "a filter that's set but whose corresponding listing data is missing does not disqualify" convention exactly (an ungeocodable Craigslist listing isn't excluded just because a radius filter happens to be set). The save-search form (`app/page.tsx`'s `handleSaveSearch`) reuses the **already-resolved** `resolvedLocation`/`radiusMiles` state from the live search filter -- no new geocoding call at save time. Deliberately **omits** `target_location`/`search_radius_miles` from the payload entirely (rather than sending `null`, unlike every other filter in that payload) when no location is currently resolved, so editing a saved search without touching the Location filter that session can't silently wipe out a radius it already had -- a real asymmetry from every other field, needed because there's no way to repopulate the zip text field on edit (no reverse geocoding) to distinguish "user wants to clear this" from "user didn't touch it." Known limitation, not solved here: a saved search's location can't be explicitly cleared via editing -- delete and recreate the search instead. Verified live against real production listing data (965 geocoded active listings): a synthetic San-Jose-centered search matched 601/965 within 10mi and 948/965 within 50mi, both directionally correct for how compact the Bay Area is; also verified via a real manual `notify.py` run post-fix that the full pipeline (staleness -> geocode -> duplicates -> deal scores -> notify) completes cleanly and sends real digest emails again.
