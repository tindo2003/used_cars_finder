# Scraper Data Quality Findings

Found 2026-07-18 by exporting `listings` to CSV and inspecting it directly (118 rows: 86 dealerinspire / Capitol Honda, 20 craigslist, 12 dealeron / Stevens Creek Toyota, 0 eBay).

## Fixed

**Duplicate rows from non-VIN sources (critical, actively growing).** 18 of the 20 craigslist rows were re-inserts of just 5 unique listings — the same `original_url` appearing 3-4 times with different `id`/`created_at`. Root cause: `scraper/main.py`'s `save_cars_to_db` upserted with `on_conflict="vin"`, but craigslist/eBay listings never have a VIN, and Postgres never treats two `NULL`s as conflicting — so every scrape run inserted a fresh duplicate instead of updating. This would keep growing unbounded on every cron run.

Related latent bug: `dealeron.py` and `dealerinspire.py` defaulted missing VINs to the literal string `"Unknown"` rather than `None`. If two vehicles on the same dealer site ever both lacked a VIN, they'd collide under `on_conflict="vin"` and silently overwrite each other.

Fix applied:
- `scraper/main.py`: upsert conflict target changed from `vin` to `original_url` (always `NOT NULL`, unique per listing across every source seen so far).
- `dealeron.py` / `dealerinspire.py`: missing VIN now stored as `None` instead of `"Unknown"`.
- `migrations/001_dedupe_listings_and_fix_conflict_key.sql`: one-time cleanup of existing duplicates + adds the unique constraint on `original_url` the new upsert relies on. **Needs to be run manually in the Supabase SQL editor** — not applied automatically.

**Bad price data from dealeron.** Two vehicles from Stevens Creek Toyota showed different wrong prices on every scrape: the Corolla Hatchback SE (VIN JTND4MBE1R3228151) went $289 → $23,505 → $66,675 across three runs; the Tundra Platinum (VIN 5TFNA5DB0SX249078) went $2,624 → $66,675 → similar drift. Root cause: `dealeron.py` read price from `data-dotagging-item-price`, a third-party analytics tag that is simply unreliable and drifts between page loads — it is not the site's actual displayed price. The real price lives in `data-pricelib`, a base64-encoded string (`"Selling Price:25888.0;...;calc_INTERNET PRICE:25973.0;..."`) that decodes to the same "Internet Price" shown on the vehicle's own detail page. Confirmed live: Corolla's real price is $25,973, Tundra's is $57,973.

Fix applied: `dealeron.py`'s `extract_price` now decodes `data-pricelib` and reads `calc_INTERNET PRICE` (falling back to `Selling Price`, then to the old analytics tag only if `data-pricelib` is missing/unparseable). Verified against the live site with `max_pages=1` — both vehicles now return their correct price.

Also added a `max_pages` parameter to both `dealeron.scrape()` and `dealerinspire.scrape()` (plumbed through `main.py`'s new `--max-pages` CLI flag), so testing a fix no longer requires paginating through the full ~370-vehicle inventory — `python main.py --dry-run --max-pages 1` checks just the first page.

**Provider signatures refactored to a shared `ScrapeOptions` dataclass.** Every `scrape()` call was growing its own ad-hoc positional argument list (`make, model, max_price, max_pages, ...`), which meant touching every provider's signature and every call site each time a new filter/option was needed. `scraper/options.py` now defines `ScrapeOptions(make, model, max_price, max_pages)`; all four providers (`dealeron`, `dealerinspire`, `craigslist`, `ebay`) take a single `options: ScrapeOptions` argument instead. Future flags (mileage, transmission, radius, etc.) just add a field to `ScrapeOptions` with a default — no provider signatures need to change.

**Most schema columns were never populated (2026-07-18).** `mileage`, `seller_type`, `transmission`, `fuel_type`, `description`, `location`, `city`, `posted_at`, `trim` were empty on every row from every source. Investigated what each source actually exposes at the search-results/list level (deliberately not visiting each vehicle's own detail page, since that would multiply scrape time per vehicle across ~10 dealers):

- **dealeron** (Stevens Creek Toyota, Fremont Chevrolet, Fremont Hyundai): the card's own data attributes are rich — added `trim` (`data-trim`), `mileage` (`data-odometer`), `transmission` (`data-trans`), `fuel_type` (`data-fueltype`), `posted_at` (`data-dotagging-item-inventory-date`), and a constant `seller_type: "dealer"`.
- **dealerinspire** (Capitol Honda/Ford/Chevrolet/Hyundai, Stevens Creek Hyundai, Sunnyvale Honda, Fremont CDJR): the `data-vehicle` JSON blob and rendered card HTML have `trim` and `fueltype` but genuinely **no mileage or transmission anywhere in the list view** — confirmed by dumping the full card `innerHTML`, the `vehicle-details` list is empty in grid view. Added `trim`, `fuel_type`, `posted_at` (`date_in_stock`), `seller_type: "dealer"`. Mileage/transmission remain null for this platform — would require an extra page load per vehicle to get from the detail page.
- **craigslist**: search results carry a `<div class="location">` (e.g. "fairfield / vacaville") — added as `city`. Checked whether listings expose an owner/dealer flag in this markup before assuming one; they don't, so `seller_type` is intentionally left null here rather than guessed.
- **city** for dealer sources comes from a new `city` field on each entry in `main.py`'s `DEALERS` list, threaded through via `ScrapeOptions.city` (added to the dataclass — this is exactly the extensibility it was built for).
- **Not populated for any source**: `description` (only on each listing's own detail page) and the PostGIS `location` (lat/long — would need geocoding a city name or per-listing address, not attempted here).

**Pagination silently not advancing, causing the same vehicles to be re-scraped dozens of times per run (found 2026-07-19).** A GitHub Actions log showed "Processed 4301 listings" in a single run while the `listings` table only had 739 distinct rows. Root cause: in both `dealeron.py` and `dealerinspire.py`, clicking "Next" can succeed with no exception thrown even when the page doesn't actually change (e.g. a dealer with only one real page of results but an always-clickable Next button) — the loop had no way to detect this and kept re-scraping the same page for the rest of its `max_pages` budget. Confirmed live: Stevens Creek Toyota has 12 real vehicles; before the fix, a bounded 20-page test scrape returned 240 entries (each of the 12 vehicles scraped exactly 20 times). With the default `max_pages=300` this could have meant re-scraping the same 12 cars up to 300 times for one dealer alone — wasted time, wasted requests against the site, and grossly inflated (but harmless, since upsert dedupes correctly) "processed" counts in the logs.

Fix applied: both providers now track the set of VINs seen on the previous page and stop the loop if the current page's VINs are identical to the previous page's, instead of trusting the "Next button exists and isn't disabled" check alone. Re-verified live: the same test now returns exactly 12 vehicles, 12 distinct VINs, and stops after detecting the first repeated page rather than running to the page cap.

**No eBay listings ever saved.** Root cause found while checking eBay's list-level markup for a location field: `https://www.ebay.com/sch/i.html` now returns **HTTP 403** to this scraper's request (confirmed directly, not merely "0 results"), so `ebay.py`'s `if res.status_code != 200: return results` silently exits every time. Likely eBay is blocking the request (missing headers/cookies, or outright bot-blocking), not a markup change. Not yet fixed.

**`original_url`'s table-wide unique constraint didn't match reality, and the workaround for it was silently losing rows (found 2026-07-20, external review).** `listings.original_url` had a table-wide `UNIQUE` constraint since migration 001, from back when it was the only conflict key. Migration 007 moved dealer-sourced upserts to `(vin, dealer_name)`, but `original_url`'s own separate uniqueness stuck around — and it doesn't actually hold: a dealer's own inventory card can legitimately link to a *different* dealer's page for the same syndicated vehicle (confirmed live: a Capitol Honda card linking to chevroletoffremont.com). The fix applied right after that discovery (see the "`original_url` can also collide across dealers" note in `memory.md`'s history) caught the resulting `23505` violation in `db.py` and skipped the row entirely rather than crashing.

That skip was worse than it looked: `duplicates.py` already has an exact-VIN-match pass built specifically to flag two dealers sharing a VIN via `duplicate_of`. But the skip happened *before* the second dealer's row was ever inserted, so that row never existed for `duplicates.py` to find — the "duplicate" was being avoided by silently losing data, not by the flagging mechanism built for it. A prior live check reporting "0 of 740 listings shared a VIN" was measuring the wrong thing: of course no collision showed up, since the colliding row was never created in the first place.

Fix applied: `migrations/012_drop_original_url_unique_constraint.sql` drops the constraint (replaced with a plain non-unique index for lookup performance). `db.py`'s `upsert()` no longer needs to catch or skip anything for this case — dealer rows (have a `vin`) keep upserting natively on `(vin, dealer_name)` unchanged, and two dealer rows sharing a VIN but different `dealer_name` now both persist and get correctly flagged by `duplicates.py`'s existing logic. VIN-less rows (Craigslist) are the only case where `original_url` was ever standing in for real identity, so that path switched to a manual lookup-then-write (read by `original_url`, update if found, insert if not) instead of relying on Postgres's `ON CONFLICT` inference, which needs a real unique arbiter that no longer exists.

**A `saved_searches` schema change (migration 014, scalar `make`/`model` → `text[]`) crashed the live scraper cron (found 2026-07-20, real GitHub Actions failure).** `scraper/runner.py`'s `run_marketplace_searches()` called `.capitalize()` directly on `search.get("make")`/`.get("model")`, assuming scalars; migration 014 made them lists, so every real run crashed with `AttributeError: 'list' object has no attribute 'capitalize'`. Root cause wasn't just the missed call site — it's that `runner.py` took `saved_searches` as untyped `Dict[str, Any]` (an explicit scoping choice from the original Pydantic rollout, see `memory.md`), so mypy had no real type to check `.capitalize()` against, and the review when building multi-make/model support only checked `notifications.py`'s usage (which *does* validate through `SavedSearch`) without grepping for every other consumer. Separately, `main.py`'s own `--dry-run` dummy fixture had independently drifted to the same stale scalar shape — meaning local `--dry-run` testing wouldn't have caught this either.

Fix applied: immediate crash fixed by iterating every make × model combination per search. Then, since the user asked to make this class of bug not happen again: `main.py` now validates `saved_searches` through `SavedSearch.model_validate()` before `runner.py` ever sees a row, and `ScrapeRunner`'s methods are typed to take `List[SavedSearch]` instead of raw dicts — reversing the part of the original Pydantic scoping decision that excluded `runner.py`. The next `SavedSearch` schema change gets caught by mypy at every call site automatically. `main.py`'s dry-run fixture was also corrected to the current real shape.

**Craigslist listings mislabeled with a fabricated make/model, and implausible prices ($0, $302-$420) winning "best deal" ranking (found 2026-07-19, from a real notification digest the user received).** Two compounding bugs:

1. `runner.py`'s `run_marketplace_searches` runs every make × model combination from a saved search's multi-select filters (documented, intentional -- see `memory.md`'s multi-make/model entry). For a "Lexus + Toyota" saved search this includes nonsense combinations that don't correspond to a real vehicle, e.g. "Toyota NX" (NX is a Lexus model) and "Lexus Camry" (Toyota makes the Camry, not Lexus).
2. `craigslist.py`'s `query=` search parameter does loose keyword matching, not an exact make+model filter -- confirmed live: searching "toyota nx" returned a Tacoma, two Camry Hybrids, and (correctly) a real Lexus NX; searching "lexus camry" returned a Lexus IS250, a Lexus RX350, and (correctly) a real Toyota Camry. The old code trusted the *search query* to label every result's make/model (`"make": make.capitalize()`) rather than checking the ad's actual title, so real listings got mislabeled as "2022 Toyota Nx" and "2020 Lexus Camry" -- confirmed by the user's actual digest email, where a "Toyota Nx" listing's link resolved to a real `toyota-camry-hybrid` Craigslist ad.

Separately, some dealer-syndicated Craigslist ads show a monthly payment ($302, $403, $420 observed) or a "$0" placeholder in the price field instead of the real price. Since `deals.ranking_key` falls back to lowest price when a make/model combination has too few comparables to score (and a fabricated "Toyota Nx"/"Lexus Camry" combination never has *any* real comparables), these bogus near-zero prices always won the "best deal" ranking and dominated the digest -- exactly what the user saw.

Fix applied to `craigslist.py`: reject a result unless the search's make and model (when set) actually appear as substrings in the ad's own title -- this rejects nonsense make×model combinations outright (searching "toyota nx" or "lexus camry" now returns nothing) and filters loose/irrelevant matches on real combinations too. Also added `MIN_PLAUSIBLE_PRICE = 500`; a real used car is never realistically sold for less, so anything under that is treated as bad source data (monthly payment or a placeholder) and skipped rather than saved. Verified live against the exact failing queries from the user's digest: "toyota nx" and "lexus camry" now return zero results; "toyota camry" and "lexus nx" both return correctly-labeled, plausibly-priced listings. 6 new tests in `tests/test_craigslist.py` (previously had zero coverage), using a `requests.get` fake since this file makes real HTTP calls directly rather than through an injected client.

Cleaned up live production data as a result: 27 already-mislabeled rows deleted directly from `listings` (a future scrape can never re-correct them, since the nonsense searches that created them will never match anything again post-fix) -- confirmed by the user before deleting.

**Two `saved_searches` rows survived migration 010's RLS rollout with `user_id = NULL`, making them invisible/unmanageable in the UI but still fully active for notifications (found 2026-07-19, same investigation as above).** The user only saw 1 saved search ("Lexus + Toyota") in "My Saved Searches" but was receiving 3 digest emails. Root cause: `saved_searches` RLS is scoped to `auth.uid() = user_id` (migration 010), which correctly hides any row whose `user_id` doesn't match the logged-in session -- `NULL` never equals a UID, so a row with no `user_id` is invisible (and unpausable/undeletable) from the frontend no matter who's logged in. Two rows ("Lexus ES", "Toyota Camry") predated migration 010's auth rollout (from the earlier no-auth/localStorage stopgap, migration 004) and were never backfilled with a real `user_id` when RLS was added. `notify.py` reads `saved_searches` via the service-role key, which bypasses RLS entirely, so it kept finding and emailing both orphaned rows regardless -- exactly the invisible "Toyota Camry" email the user was seeing. A full-table check confirmed these were the only 2 rows anywhere with `user_id IS NULL`, so this was a one-time leftover from the auth migration rather than an ongoing gap (new saved searches always get a real `user_id` at creation, per the Auth decision in `memory.md`). Deleted both after confirming with the user.

**One dealer timing out crashed the entire scheduled scrape, silently skipping every dealer after it (found 2026-07-19, real GitHub Actions failure).** `dealeron.py`'s initial `page.goto(..., wait_until="networkidle")` hit Playwright's default 30s timeout on `chevroletoffremont.com` and raised `TimeoutError` -- the same "networkidle hangs on sites with continuously-polling chat widgets/trackers" issue `dealerinspire.py` and `dealersocket_gemini.py` had already hit and fixed, just not yet applied to `dealeron.py`. The real problem was broader than one dealer's flakiness, though: `runner.py`'s `run_dealer_scrapes()` had zero error handling around each dealer's scrape call, so this single unhandled exception propagated all the way up through `main.py` and killed the whole process -- with 16 dealers now in `DEALERS`, any one of them being slow, down, or bot-walling the scraper for a moment takes out every dealer after it in that run, not just the one that failed.

Fix applied: `dealeron.py`'s initial navigation switched from `wait_until="networkidle"` to `"domcontentloaded"`, matching the other two providers -- the existing inventory-container probe loop was upgraded from an immediate (non-waiting) `query_selector` check to `wait_for_selector(selector, timeout=15000)` per candidate, since the immediate check relied on `networkidle` having already settled the page and would otherwise false-negative on a slower-to-render but perfectly working site. Separately (and more importantly for the systemic risk), `runner.py`'s `run_dealer_scrapes()` now wraps each dealer's scrape+save in a try/except: a failing dealer is logged and counted in a new `progress["dealer_errors"]` (surfaced in `main.py`'s final summary line) instead of crashing the run, so every other dealer still gets scraped that day. Verified live against the exact dealer from the crash (chevroletoffremont.com, 46 vehicles parsed correctly) and with a new test in `tests/test_runner.py` asserting a raising dealer doesn't stop subsequent dealers from being scraped and saved.

**A real (non-dry-run) scrape/notify run never exits the process on its own, meaning every scheduled run on ECS Fargate would bill indefinitely until manually stopped (found 2026-07-20, first real production Fargate run after deploying).** The scheduled `scrape` task logged its final `"Done. Processed ... listings"` line and then produced zero further output, but the ECS task stayed `RUNNING` indefinitely -- confirmed by manually calling `aws ecs stop-task`, which succeeded with a clean `exitCode: 0`, meaning the process really had finished its work and just wasn't exiting on its own. Reproduced locally too: a real (non-`--dry-run`) local run hangs the same way after its last print, while every `--dry-run` run this whole project has ever done exits cleanly -- because `--dry-run` never calls `get_supabase()` at all, so this code path (the one every scheduled production run actually takes) had never really been exercised end-to-end before going live on AWS. Root cause is almost certainly a non-daemon thread the Supabase client leaves running in the background (its `realtime`/`websockets` dependency is the prime suspect, though not confirmed by direct inspection) -- CPython refuses to fully exit until every non-daemon thread finishes, and `sys.exit()` doesn't override that.

This also explains a side puzzle from the same investigation: the CloudWatch logs for the hung task showed every single line at one identical timestamp, which looked like the task had frozen from the very start. It hadn't -- Python fully buffers stdout by default whenever it isn't attached to a real terminal (true both in a Docker container and when piped locally), so the entire run's output was sitting in memory and only got flushed as one batch. Re-running locally with `python3 -u` (unbuffered) confirmed real, steady progress the whole time; the "frozen since the start" appearance was purely a visibility artifact, unrelated to the actual hang (which was real, just at the *end* of the run, not throughout it).

Fix applied: both `main.py` and `notify.py` now call `sys.stdout.flush()` followed by `os._exit(0)` at the very end of their `__main__` blocks, forcing immediate process termination regardless of any lingering non-daemon threads. Verified locally: a real, non-dry-run run now exits cleanly right after its final print (previously required a manual kill).

**Fargate re-verification was inconclusive, not confirmed either way, 2026-07-20 same night.** Attempted to re-verify by pushing the fixed image and manually running the `scrape` task again -- but by that point the EventBridge schedules (created during the same `terraform apply`) had already started firing on their own */15 cadence, so **3 tasks ended up running concurrently** (the manual test plus 2+ auto-triggered ones) with their CloudWatch log output interleaved in the same log group, making it impossible to tell which task's output was which. Rather than let a genuinely unresolved reliability question (does the fix actually work under Fargate specifically, not just locally) run unattended overnight, all 3 running tasks were stopped and **both EventBridge schedules were manually set to `DISABLED`** (via `aws scheduler update-schedule`, outside of Terraform -- `terraform apply` will detect this as drift and re-enable them next run unless the config is updated first). **Re-verify properly next session**: re-run just one task at a time (nothing else scheduled or running concurrently) and confirm it reaches `STOPPED` with `exitCode: 0` on its own before re-enabling the schedules.

## Testing recommendation

This investigation only happened because of a manual CSV export. A layered automated approach would catch this class of bug going forward:
1. **Fixture-based unit tests** per provider — save a real HTML/JSON sample from each site, assert `extract_vehicle_data` parses expected fields correctly. Fast, deterministic, no network calls, catches parsing regressions (would have caught the price bug if a fixture with that vehicle's markup existed).
2. **Data sanity checks before save** — price bounds relative to model year, required non-null fields, flag or reject outliers rather than silently writing them. Would have caught the $289 Corolla immediately.
3. **Dedup/upsert correctness test** — assert that saving the same listing twice with the same `original_url` updates rather than inserts a second row.
4. **Route every consumer of a DB-backed shape through its Pydantic model, not just the one you're actively editing** — the `saved_searches.make`/`model` schema change (2026-07-20) broke `runner.py` because it read `saved_searches` as an untyped dict, so mypy had nothing real to check when the column type changed. `grep -rn "search\.\(get\|make\|model\)"` (or similar, for whatever field is changing) across the whole codebase before considering a schema-affecting change complete would have caught this before it shipped, not after a real cron failure.
